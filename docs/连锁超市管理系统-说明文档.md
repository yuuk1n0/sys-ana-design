# 连锁超市管理系统说明文档

## 1. 文档目标

本文档用于在当前 `vue-fastapi-admin` 项目基础上，明确连锁超市系统的三部分建设路径，并聚焦第一部分“店长管理个人门店”可直接落地的模块范围。

## 2. 现有项目基线能力

当前项目可直接复用为系统底座：
- 账号认证与鉴权：JWT、RBAC、菜单权限、接口权限
- 管理能力：用户、角色、菜单、部门、审计日志
- 技术实现：FastAPI + Tortoise ORM + Vue3 + Naive UI

结论：现有项目适合承载“门店经营后台”的平台层能力，业务域按模块增量扩展即可。

## 3. 业务闭环与三部分拆分

整体目标仍是“商品建档—采购入库—门店调拨—POS 销售—结算对账—经营分析”闭环，实施时拆成三部分：

### 第一部分（当前实现范围）

聚焦每个门店的日常经营基础能力，只做店长管理个人门店所需模块：
- 商品管理
- 门店库存
- 收银（POS）
- 盘点
- 日结

当前明确不展开的场景：
- 采购入库
- 门店调拨
- 结算对账的复杂财务场景

### 第二部分（后续扩展）

- 采购入库全流程
- 门店间调拨
- 供应商协同与到货跟踪

### 第三部分（后续扩展）

- 总部级经营分析
- 多门店对比看板
- 结算对账与利润分析

## 4. 第一部分系统定位

第一部分定位为“单门店运营系统”，强调店长对个人门店的可控、可查、可追溯：
- **可控**：商品状态、库存阈值、盘点处理、日结确认
- **可查**：销售流水、库存流水、盘点差异、日结汇总
- **可追溯**：关键变动具备单据与日志链路

## 5. 第一部分功能模块化设计

## 5.1 M1 商品管理模块

核心目标：保障门店可售商品的准确与可维护。

功能边界：
- 商品主档：名称、分类、条码、单位、零售价、状态
- 门店商品：门店上架、下架、停售、门店售价
- 查询能力：按名称/条码/分类检索

输出结果：
- 可被 POS 快速检索并开单
- 商品状态对库存和收银实时生效

## 5.2 M2 门店库存模块

核心目标：店长实时掌握库存，减少缺货与异常。

功能边界：
- 库存余额：按门店 + SKU 查询
- 库存流水：销售扣减、退货回补、盘点调整
- 预警机制：低库存预警、负库存拦截策略

输出结果：
- 每一笔库存变化可追溯来源
- 店长可按阈值识别补货风险

## 5.3 M3 收银（POS）模块

核心目标：完成门店前台销售与退货的基础闭环。

功能边界：
- 快速开单：扫码/检索商品、修改数量、计算应收
- 支付记录：记录支付方式与金额
- 销售退货：基于原单退货并联动库存

输出结果：
- 形成销售单、销售明细、支付记录
- 自动更新库存与销售统计

## 5.4 M4 盘点模块

核心目标：校准账实库存差异，提升库存准确率。

功能边界：
- 盘点单：创建、录入实盘数、确认差异
- 差异处理：盘盈入账、盘亏扣减
- 历史追溯：盘点记录查询

输出结果：
- 盘点完成后库存余额同步修正
- 盘点差异纳入审计与经营分析

## 5.5 M5 日结模块

核心目标：完成门店营业日的经营归档与结算汇总。

功能边界：
- 日结汇总：销售额、订单数、退款额、支付方式分布
- 班次核对：可按收银员维度汇总
- 关账动作：日结确认后锁定当日统计口径

输出结果：
- 店长每日可查看门店经营摘要
- 为后续总部分析提供标准化日数据

## 6. 第一部分角色与权限

- **店长**
  - 管理本门店商品、库存、盘点、日结
  - 查看本门店 POS 销售与报表
- **收银员**
  - 仅执行收银与退货操作
  - 可查看本人班次相关信息

权限原则：
- 全部业务数据按 `store_id` 强隔离
- 店长不可访问其他门店数据
- 关键操作写入审计日志

## 7. 第一部分数据模型建议

建议新增实体（第一部分最小集合）：
- 商品域：`product`, `product_category`, `store_product_price`
- 库存域：`store_inventory`, `inventory_txn`
- 销售域：`pos_order`, `pos_order_item`, `payment_record`, `sales_return_order`
- 盘点域：`stocktake_order`, `stocktake_item`
- 日结域：`store_daily_close`, `cashier_shift_summary`

## 8. 第一部分验收口径

- 商品可建档并在 POS 检索销售
- 销售后库存自动扣减，退货后库存自动回补
- 盘点差异可处理并回写库存
- 店长可完成按日关账并查看日结报表
- 店长账号仅可访问个人门店数据

## 9. 实施建议

- 先做“单门店闭环”，再做“多门店协同”
- 先保证数据正确，再追求流程复杂度
- 优先实现统一单据与库存流水模型，为第二、三部分扩展留接口
